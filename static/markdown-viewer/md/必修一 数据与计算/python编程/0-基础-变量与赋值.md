# 0-基础-变量与赋值

## 一、什么是变量？

变量是计算机内存中的一个存储位置，用于保存数据。在编程中，变量用来表示程序中的数据，其值可以在程序执行过程中发生变化。

### 1.1 变量的特性

- **名称唯一:** 每个变量都有唯一的名称。
- **存储数据:** 变量用来存储数据。
- **值可变:** 变量的值在程序运行时可以改变。

------

## 二、变量的命名规则

在定义变量时，必须遵守以下命名规则：

1. **字母、数字和下划线组合:** 变量名只能包含字母 (a-z, A-Z)、数字 (0-9) 和下划线 (_)。

2. **不能以数字开头，并且区分大小写:** age 和 Age 是两个不同的变量。

3. **不能使用保留字:** 如 if, else, while 等。

   > 保留字是指在编程时的一些特殊的关键词，类比一下的话就好像英文中的is、am、are、by等用于标记句子结构的单词。一旦用保留字作为变量名，将导致程序的解读时出现歧义。

  

- **好习惯：有含义的变量名** 

  > 很多时候，我们使用描述性的变量名，如 score 表示分数、count表示计数、answer表示结果，而在编程中，也有很多人喜欢用三个字母的简写，如sco表示分数，cnt表示计数，ans表示结果等。

------

## 三、赋值语句

赋值语句用于将一个值存储到变量中，格式如下：

变量名 = 值


> 如此代表将右边的值存入左边的变量

### 示例:

```python
## 定义一个变量并赋值
age = 16
name = "Alice"
score = 95.5
```

### 多变量赋值:

```python
## 同时给多个变量赋值
x, y, z = 10, 20, 30
```

### 【案例】python中的变量交换
将a和b两数进行交换
#### 方法一：临时变量
```python
a = 1
b = 2
c = a #设置临时变量用于交换
a = b
b = c
print(a, b) # 可以看到，ab已经交换
```
#### 方法二：直接复制
```python
a,b = b,a
```
这也是最常用的方式

#### 方法三：相加
```python
a = a + b #先令a为两数之和
b = a - b #如此，则b变为原来的a
a = a - b #a变为原来的b
```

## 四、扩展阅读

#### 驼峰命名法

> 驼峰命名法（CamelCase）是一种在编程中给变量、函数、类等命名时常用的写法风格。它的特点是在名字中不使用空格或者下划线，而是透过单词的首字母大小写变化让名字看起来像一只骆驼的背上有“峰”一样的起伏。想象一下，一只骆驼背上的驼峰高低起伏，而单词的开头字母在一个词中也会有大有小，“大小写”就像这只骆驼的“峰”——这就是驼峰命名法的由来。

比如说，你想表达一个变量是用来存放学生的姓名。如果你只写 studentname，从头到尾小写，读起来有点费劲，不好分辨其中的单词边界。而如果采用驼峰命名法，就有两种常见形式：

1. **小驼峰式**（lowerCamelCase）：第一个单词首字母小写，其后每个新单词的首字母大写。例如：studentName。这里的 s 是小写，然后 N 大写，让人一眼看出两个单词“student”和“Name”。
2. **大驼峰式**（UpperCamelCase 或 PascalCase）：每个单词的首字母都大写。例如：StudentName。看到 S 和 N 都大写，能很快分辨出这是两个独立的词组合而成的一个命名。

这种命名法的好处就是在不依赖下划线的情况下，将变量名或函数名的结构清晰地呈现出来，让代码更加易读。这就像给你的代码加上了一个小小的“变形金刚滤镜”：统一规整，又充满规律性。当你在阅读代码时，眼睛会很自然地根据字母的大小写变化，将连续的字符切分成多个词，理解起来更加轻松。

总结一下：**驼峰命名法就是在命名时，通过单词首字母的大小写变化，让变量名看起来像骆驼背上的“驼峰”一样，有起有伏，以此达到更高的可读性。**

#### 程序注释

当然，在程序员群体中，并不是所有人都能完整地遵守变量的命名约定，这让程序员在交接工作时变得相当痛苦。

比如说，某程序员在编写时可能用xm表示姓名，用age表示年龄，用sfz表示身份证，这样的中英文拼音混用，对代码理解是一种负担，有时连编写者自己都会忘记变量的含义，为了解决这个问题，程序设计了注释功能。

> 在 Python 中，#号表示注释
>
> 注释是帮助你解释代码含义、用途或者特别逻辑的文字说明，它们不会被程序执行，而是专门写给“人”看的。换句话说，注释就像在代码里贴的一张小纸条，在纸条上写明了这个地方的代码到底在干什么、为什么这样写，方便你日后或者其他人再次阅读代码时，不用凭空猜测。

```python
a = 39  # a表示学号
b = "fly" #b表示姓名
```

然而，注释这种“说明书”式的存在，并不总是能挽回糟糕的命名所带来的麻烦。有些程序员对注释的态度就像给草坪修剪杂草一样——他们会在刚刚开发完一个功能时写下简短几行注释，解释这个函数、这个变量的用途和逻辑。但随着时间推移，代码不断演化，功能增加、逻辑修改，最初的注释却常常无人问津，渐渐变成了过期的“历史文物”。

于是，我们就进入了一个奇妙的循环：

1. **初识阶段**：程序员像给宝贝一样，为新代码精心添加合适的命名和清晰的注释。此时的代码读起来就像一本绘本，一切信息都井然有序。
2. **中年危机**：几个月后，新进的同事继承了这份代码。受时间和压力所迫，他们迅速加入新特性，而顾不得重构和完善命名规则，仅仅在注释中草草记录改动，使得变量名与注释脱节。
3. **混乱时代**：再过一段时间，当又一位接手这段代码的程序员到来，看到的是一片名不副实的“变量丛林”和一堆与现实不符的注释。“xm”究竟是姓名？还是某个神秘的内部标识？为什么明明叫“age”的变量却用来储存入职时间？而注释里还一口咬定这是“用户注册天数的统计”？

而在实际工作中，为了拯救这片混沌的“代码荒漠”，团队或许需要制定更严格的规范——不仅要规定命名方式，还要定期检视注释和代码的匹配度。在敏捷开发和持续集成的时代里，良好的代码审查（Code Review）和定期的重构（Refactoring）就显得至关重要。

总而言之，虽然注释能够在一定程度上缓解命名不当的困境，但从长远来看，命名清晰、风格统一的代码才是最好的注释。只有当变量名、函数名本身就能描述清楚它们所代表的含义和用途时，注释才能成为锦上添花的额外说明，而非用来抵消命名不当所带来的混乱。
